# Load monthly files from config (2026 monthly exports)
if config:
    monthly_configs = config.get('sources', {}).get('monthly', [])
    for item in monthly_configs:
        if isinstance(item, dict):
            path = item.get('path', '')
        else:
            path = item
        
        if path and Path(path).exists():
            # Use 2026 as year for monthly files
            file_configs.append((path, 2026, None))
            logger.info(f"  Added monthly file: {Path(path).name}")
        elif path:
            logger.warning(f"  Monthly file not found: {path}")

=================================================
LESSONS LEARNED
=================================================

1. CONFIGURATION VS IMPLEMENTATION
   - Having config entries doesn't mean code uses them
   - Always verify code actually loads configured values
   - Comments like "now loaded from config" can be misleading

2. DATE RANGE HARDCODING
   - Hardcoded date ranges can silently filter expected data
   - Consider using dynamic date ranges or config-driven dates
   - Test with future dates to catch filter issues

3. INCREMENTAL MODE ASSUMPTIONS
   - Baseline file format matters (raw vs polished)
   - Document expected baseline file format clearly
   - Add validation to detect polished files in incremental mode

4. VERIFICATION IS CRITICAL
   - Always verify output date ranges match expectations
   - Check record counts against expected totals
   - Sample data across different years to ensure coverage

5. PARALLEL PROCESSING BENEFITS
   - 8-worker parallel loading completed 12 files in ~2 minutes
   - Memory optimization (24.5% reduction) helps with large datasets
   - Chunked reading for files >50MB prevents memory issues

=================================================
TECHNICAL NOTES
=================================================

1. Monthly file loading supports two config formats:
   - Dict format: {'path': '...', 'month': '2026-01', 'notes': '...'}
   - String format: 'C:\\path\\to\\file.xlsx'

2. Date filtering is applied at three stages:
   - Load time (implicit, based on END_DATE)
   - Post-concatenation filter (explicit)
   - Keeps records where START_DATE <= TimeOfCall <= END_DATE

3. Parallel loading uses ThreadPoolExecutor:
   - Max workers configurable (default: 8)
   - Can be disabled in config for debugging
   - Results sorted by year after loading

4. Memory optimization:
   - Low-cardinality strings converted to categorical
   - Numeric types downcasted (int64→int32, float64→float32)
   - Applied after concatenation, before export
